排程備份產品化 · 專項增強計畫

僅涵蓋「排程備份」由 dry-mock 升級為可營運真實週期備份。 不含拓撲、diff、retention 本體、新廠牌等其他 epic。 本文件為 PLAN；未獲「start building」前不實作。

基準分支 AAA-addon 現況 dry-mock（P2 skeleton） MVP Phase A + B + C 核心 語言 繁體中文 實作計劃稿 .hermes/plans/2026-07-23-scheduled-backup-productization.md
警告（現況）：側欄「排程備份」目前不會 SSH、不會打 Agent、不會寫快照。 僅解析 CSV 並更新 last_run_at／last_result。請勿當成已在自動備份設備。

1. 範圍與目標

1.1 In scope

  • 排程定義持久化（沿用／擴充 store/schedules.db）
  • 模式：dry_mock｜live
  • SSH 憑證安全存放與綁定（至少每排程一組帳密）
  • Watcher 到期與「立即執行」觸發真實備份（走既有 runner）
  • 防重入、單 Portal flock、執行歷史、審計
  • UI／Handbook／README 產品化敘述

1.2 Out of scope（本專項不做）

  • 鄰居拓撲（已移除）、Config Diff、快照保留功能本體
  • 新廠牌 parser、CMDB 整合、完整通知中心（可列 Phase D）
  • K8s 多副本 leader election（僅單實例 + flock 契約）
  • 重寫 NetDriver／批次備份 runner 核心邏輯

1.3 成功樣貌（MVP）

營運者建立 live 排程 → 綁定加密憑證 → 啟用 → 到期或手動觸發 → 與「批次備份」同等品質寫入 store/ 與 index.db → 可在排程頁／審計看到 ok/fail 與 job／run id → 誤開風險可控（預設 mock、confirm）。

2. 現況（dry-mock）

元件路徑行為
排程核心 nccm/backup/schedule.py CRUD、every_minutes、enable、watcher ~30s、mock_run_csv
Web web/main.py /schedules*、schedules.html admin／operator；建立、立即 mock、toggle、delete
真備份（未接） runner.py、job_manager.py 批次備份已用；排程尚未呼叫
資料 store/schedules.db name、csv_text、間隔、last_run、last_result；無密碼、無 mode
文件 Handbook #use-schedules 已標註 dry-mock

3. 缺口（產品化必須補）

缺口風險／影響
只 mock 不備份誤以為已自動備援
無 SSH 憑證無法 live
CSV 一組帳密多廠牌需拆排程（與批次相同限制）
無 flock／leader多 Portal 可能雙跑
job 僅記憶體重啟丟進度；難關聯歷史
無防重入長備份與短間隔重疊
無 dry／live 開關易誤觸全網
無結構化 run 歷史營運難追溯

5. 目標架構

Portal (single replica)
  schedule watcher (30s + store/schedule.lock)
       │ due?
       ▼
  schedule_executor
       ├─ mode=dry_mock → mock_run_csv → schedule_runs
       └─ mode=live
            ├─ decrypt credentials (Fernet)
            ├─ load_devices_csv
            ├─ start_backup_job_async  ──► job_manager ──► run_backup_job
            │                                    │              │
            │                                    │              ▼
            │                                    │         NetDriver Agent → SSH
            │                                    │              │
            └─ on complete: runs + audit + last_*◄┘              ▼
                                                         store/ + index.db

5.1 原則

  • 不重寫備份：一律 run_backup_job／start_backup_job_async
  • Cisco enable 規則維持 runner 既有邏輯
  • Agent URL 與批次備份相同（NCCM_NETDRIVER_URL）
  • 密碼不進 git、不進 SSE／job log 明文

5.2 資料模型（摘要）

schedules 擴充

mode（dry_mock｜live）、username、password_enc、enable_password_enc、 running_job_id、last_job_id、last_ok_count、last_fail_count、created_by…

schedule_runs（新表）

schedule_id、started/finished、mode、job_id、run_id、status、ok/fail/device_count、summary、triggered_by

舊資料

既有列 migration 後 mode=dry_mock，行為與今日相容。

APhase A · 資料與加密（仍可不連設備）

前置

  • 決定 NCCM_SECRETS_KEY 產製與 compose secret 掛載方式
  • cryptography（Fernet）納入依賴
  • 無需 lab 設備（本階段可不 SSH）

交付物

  • nccm/backup/secrets.py 加解密
  • schema 幂等 ensure（mode、密文字段、schedule_runs）
  • CRUD／UI：模式、憑證欄（空白=保留原密文）
  • 無 key 時拒絕建立／切換 live
  • 審計：schedule create/update/delete
  • pytest：加解密、migration、預設 dry_mock

完成定義

不開 live 時與現況 mock 相容；DB 可安全存密文；測試綠。

BPhase B · Live 執行串 runner

前置

  • Phase A 完成
  • lab 或 SimuNet：Agent 可達 + 至少 1 台 CSV 設備（手動 UAT）
  • 單元測試以 monkeypatch runner，不強迫 CI 連實機

交付物

  • schedule_executor.py：dry／live 分支、triggered_by
  • 整合 start_backup_job_async；完成輪詢或 on_complete callback
  • D5：同 schedule running → skip + audit
  • store/schedule.lock 包住 tick
  • UI：「立即真實備份」+ confirm；「立即 dry-mock」保留
  • last_result 含 ok/fail、job_id、run_id
  • 審計：run_start／run_finish／skipped
  • pytest：mock runner 全路徑；缺憑證不呼叫 runner

完成定義

Live 路徑可在 lab 產出新快照；重入與 lock 行為有測試。

7. 測試與 UAT

自動化（CI 預設）

  • Fernet roundtrip；無 key 拒 live
  • due／skip running／缺憑證
  • executor + monkeypatch run_backup_job／async
  • flock 第二 tick 不雙跑（tmp lock）
  • pytest -m "not private" 全綠

手動 UAT（lab）

  1. docker compose up；設定 NCCM_SECRETS_KEY
  2. 建 dry_mock 排程 → 立即 mock → 確認不新增快照
  3. 改 live + 帳密 → 立即真實備份 → inventory 有新版本
  4. 短間隔 + 長備份 → 第二次 skip
  5. admin 審計可見 schedule_run_*
  6. viewer（若實作）只讀
禁止將真實密碼、lab 明文 dump 提交 git。

8. MVP 驗收標準

  1. dry_mock 行為與產品化前相容（解析 CSV、不 SSH）。
  2. 設定 secrets key 後可存 live 憑證；UI／API／DB 無密碼明文。
  3. 手動與 watcher 皆可觸發與批次同等備份；成功可見新快照。
  4. 同排程不平行雙跑 live。
  5. run 結果可區分 ok/fail；審計有對應事件。
  6. 無 key 無法啟用 live（錯誤清楚）。
  7. pytest 公開集全綠；Handbook／README 已更新。

9. 風險與緩解

風險緩解
誤開 live 打全網預設 dry_mock；confirm；可預設 enabled=0
密碼進 logcode review 檢查點 + 測試
備份時長 > 間隔skip if running
Agent 不可達runner 錯誤 → runs failed + audit
多 Portal 雙跑flock + 文件單副本契約
與 per-vendor 密碼期待落差MVP 文件寫清；Phase D map
